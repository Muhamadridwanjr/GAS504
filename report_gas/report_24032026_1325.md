## 🔧 Bagian 3 — Implementasi yang Dikerjakan

### P0.1 — Circuit Breaker ✅
**File dibuat**: `gas-web-backend/src/services/circuit_breaker.py`

Fitur:
- State machine: `CLOSED → OPEN (5 failures) → HALF_OPEN (60s) → CLOSED`
- Auto-fallback ke Kimi AI saat circuit OPEN
- Half-open probing: max 2 request saat recovery test
- Endpoint monitoring: `GET /api/v1/system/circuit-breakers`

```
Behaviour:
  5 consecutive failures → circuit OPEN → all requests blocked + Kimi fallback
  After 60s → HALF_OPEN → 2 probe requests allowed
  Probe success → CLOSED (normal)
  Probe fail → back to OPEN
```

**File dimodifikasi**:
- `gas-web-backend/src/api/v1/analysis.py` — `_call_core()` sekarang pakai circuit breaker
- `gas-web-backend/src/api/v1/health.py` — tambah CB status di system health

---

### P0.2 — Atomic Credit Deduction ✅
**File dimodifikasi**: `gas-web-backend/src/api/v1/billing.py`

Fungsi baru:
- `reserve_credits(user_id, cost)` — Lua script atomic: check + deduct + set reservation TTL 120s
- `confirm_reservation(reservation_id)` — hapus reservation key (kredit sudah terkonfirmasi)
- `release_credits(reservation_id)` — rollback: kembalikan kredit ke user + hapus reservation

Pattern diimplementasikan di `_call_core_with_credits()`:
```
SEBELUM (BROKEN):
  deduct 3cr → call strategy-core → TIMEOUT → kredit hilang ❌

SEKARANG (FIXED):
  reserve 3cr → call strategy-core → SUCCESS → confirm ✅
                                  → TIMEOUT  → release (refund) ✅
                                  → ERROR    → release (refund) ✅
```

Redis keys:
- `user:{user_id}:credits` — balance
- `credit_reserve:{uuid}` — reservation dengan TTL 120s (auto-expire kalau server crash)

### P0.3 — Redis AOF Persistence ✅
**File dimodifikasi**: `docker-compose.yml`

Perubahan:
```yaml
# SEBELUM:
command: redis-server --maxmemory 256mb --maxmemory-policy allkeys-lru

# SESUDAH:
command: >
  redis-server
  --maxmemory 512mb
  --maxmemory-policy allkeys-lru
  --appendonly yes        # AOF enabled
  --appendfsync everysec  # flush tiap 1 detik
  --save 60 1             # RDB snapshot tiap 60s
  --save 300 10
  --dir /data
volumes:
  - gas-redis-data:/data  # persistent volume
```

Volume baru: `gas-redis-data` ditambahkan di `volumes:` section docker-compose.

### P1.1 — Dynamic Quant Engine by Regime ✅
**File dimodifikasi**: `gas-quant-orch/src/core/orchestrator.py`

Regime-to-engine mapping:
```python
REGIME_ENGINES = {
    "TRENDING":  ["regime", "trend", "pattern", "orderflow"],   # 4 engines
    "RANGING":   ["regime", "statarb", "phase"],                 # 3 engines
    "BREAKOUT":  ["regime", "orderflow", "trend", "pattern"],    # 4 engines
    "MOMENTUM":  ["regime", "orderflow", "trend", "pattern"],    # 4 engines
    "UNKNOWN":   ALL_ENGINES,                                    # 6 engines (fallback)
}
```

2-Phase execution:
1. **Phase 1**: Quick regime check dengan asyncio.wait_for timeout 0.5s
2. **Phase 2**: Run hanya engines yang relevan untuk regime tersebut

Hasil test: `RANGING` → hanya 3 engines dijalankan (hemat 3 HTTP calls)

Response sekarang include:
- `regime_used`: regime yang terdeteksi
- `engines_run`: list engines yang benar-benar dijalankan
